1プロジェクト概要

複数台の Arduino が PCに依存せず自律的に同期 し、合奏（オーケストラ）を成立させるシステム。 指揮棒の動き（加速度センサ）でテンポと演奏開始を制御し、カエルの歌を輪唱（カノン）として奏でる。

社会的背景：聴覚に障害のある人にも音楽を楽しんでもらうため、音に連動して動く サーボ（カエルの折り紙）と、Processing 画面の波形表示で「音を視覚化」する。

担当楽器

主旋律3台＝バイオリン・トランペット・オルガン／リズム1台＝ギロ。本書はバイオリン子機とサーボ子機を扱う。

本実装の役割

同期親機から来る合図に従い、メロディ・テンポ・輪唱を生成して Processing（音）とサーボ（視覚）を駆動する。

守る制約

Processing の音色合成クラスは変更しない。子機がメロディ・テンポ・輪唱を担い、Processing は発音と波形表示のみ。

2システム全体構成

センサ→同期親機→各子機の一方向の流れ。同期親機 SyncMain 自身が WiFi アクセスポイントになり、 子機（センサ・楽器・サーボ）がそこへ無線接続する。

制御・センサ 同期（親機＝AP） 楽器（音） サーボ（視覚）

※「Arduino 1台 ＝ PC 1台」：各楽器子機は USB で自分専用のPCの Processing に繋ぐ。 USB（音符送出）と WiFi（同期受信）は独立して同時に動くため、シリアルポートの取り合いは起きない。

3通信プロトコル（同期側コードに準拠）

同期側の実コード（SyncMain／InstClient）から読み取った正式仕様。 楽器・サーボ子機はこれに厳密に合わせている。

3-1. 接続（WiFi＋TCP登録）

項目値
親機SSID / PASSH1-SyncAP / sync2026
親機IP192.168.4.1（AP）
TCPポート（登録用）9000
UDPポート（合図用）9001（ブロードキャスト 192.168.4.255）
子機名sens / inst1 / inst2 / inst3 / inst4

登録手順：子機が TCP 接続 → 自分の名前を送信 → 親機が WELCOME 返信 → 全子機が揃うと READY を一斉送信。

3-2. 実行時の合図（UDPブロードキャスト）

メッセージ意味送信
START演奏開始3回重複送信（取りこぼし防止）
LEVEL:1 / LEVEL:2 / LEVEL:3 テンポ段階（速度）変更のたびに3回

センサ子機 → 親機（TCP）は "START" と "1"/"2"/"3"。親機がそれを UDP の START / LEVEL:x に変換して全子機へ中継する。

重要：プロトコルにあるのは START と LEVEL（3段階）だけ。 「輪唱だけを開始する合図」や「停止」は存在しない。以前検討した A/B の2合図やテンポ5段階は実コードと不一致のため破棄した。

4楽器子機 InstClient（バイオリン）

チームの InstClient に演奏処理を統合。ネットワーク層（Instfunc / InstClass）は無改修で再利用し、 InstClient.ino に「メロディ送出・テンポ・輪唱」を追加した。

動作の流れ

状態処理
ready < 2connection() で TCP 登録（WELCOME/READY）し、recieveStart() で START を待つ
START 受信（ready==2）startPlaying()：先頭から再生開始
演奏中recieveLevel() でテンポ即時変更＋stepSequencer() で音符送出

中核コード（音符送出）

void stepSequencer(){
    if (!playing) return;
    unsigned long now = millis();        // delay()ではなくmillisで刻む
    if (now - lastStepTime >= (unsigned long)tempoMs) {
        lastStepTime = now;
        int durMs = (int)(tempoMs * 0.8);            // ゲート率80%
        sendVoiceNote(playPos, durMs);               // 声部0: 主旋律
        sendVoiceNote(playPos - ROUND_OFFSET, durMs);// 声部1: 輪唱(16ステップ遅れ)
        playPos++;                                    // 剰余でループ＝無限カノン
    }
}
// 指定位置の音符を "Hz,ms" でUSB→violin.pde へ(休符・未開始は送らない)
void sendVoiceNote(long pos, int durMs){
    if (pos < 0) return;
    float pitch = melody[pos % melodyLength];
    if (pitch <= 0.0) return;
    Serial.print(pitch); Serial.print(","); Serial.println(durMs);
}
テンポの途中変更：millis() で刻むので LEVEL を受けて tempoMs を書き換えると、次のステップから即座に新しいテンポになる（曲の途中でも切り替わる）。
シリアル共用について：Instfunc のデバッグ出力も USB に流れるが、 violin.pde は「数値,数値」以外の行を無視するので、音符行と混在しても問題ない。

5輪唱のしくみ（1台で2声部）

輪唱は「楽器ごとに声部を分ける」のではなく、1台の中でカエルの歌を自分自身とずらして重ねて作る。 全楽器が同じ輪唱を同時に演奏する。声部1は1フレーズ＝16ステップ（＝8分音符16個＝2小節）遅れで自動的に入る。

各ステップで「声部0＝melody[playPos]」と「声部1＝melody[playPos-16]」を両方送る。 violin.pde は受信音符ごとに新しい音を重ねて鳴らせる（ポリフォニック）ので、1台だけで輪唱に聞こえる。

同期プロトコルに「輪唱開始の合図」が無いため、輪唱は START 後に自動で入る設計。 指揮者が任意のタイミングで2声目を入れたい場合は、親機に新メッセージ（例 ROUND）の追加が必要。

6テンポ（3段階）

親機の LEVEL:1/2/3 を bpm に対応づける。8分音符の長さ tempoMs = 30000 / bpm。

LEVEL意味（暫定）bpm8分音符(ms)
1遅い100300
2普通（既定）140214
3速い180167
const int bpmTable[4] = {0, 100, 140, 180}; // index = level(1..3)
tempoMs = 30000 / bpmTable[level];
要調整：センサ側の「速度→レベル」変換ロジックは未実装（福島さん担当）。 どの数字が速い/遅いかの規約が決まったら bpmTable を合わせる。5段階に戻すなら、センサが 1〜5 を送り 親機 recieveFrag() がそれを通すよう センサ／同期側の改修が必要。

7サーボ子機 ServoClient（視覚フィードバック）

楽器子機と同じ曲・輪唱・テンポで動き、鳴っている音階に対応するサーボを動かす。 カエルの歌は ド・レ・ミ・ファ・ソ・ラ の6音なので、サーボ6機に1対1で割り当てる。

音階ドレミファソラ
周波数(Hz)523.2587.3659.2698.4784.0880.0
サーボ端子D2D3D4D5D6D7

2声部の輪唱を反映するので、最大2機が同時に動く。発音時に ACTIVE_ANGLE(70°) へ、 発音時間が過ぎたら REST_ANGLE(0°) へ非ブロッキングで戻る。

受動リスナー設計：サーボは TCP 登録をせず、AP に接続して UDP を聞くだけ。 理由は、親機 SyncMain の子機枠が {sens, inst1..inst4} の5枠固定で空きが無いため。 START は3回ブロードキャストされ UDP は登録不要で受信できるので、起動時に接続しておけば取りこぼさない。 （登録child化したい場合は親機側の配列拡張が必要。）

8Processing 側（音色・波形）

Processing は音色合成・発音・波形表示のみを担当し、音階やテンポは子機（ino）から受け取る。

  • 受信：シリアルで "Hz,ms" を受け取り、その音程・長さで発音（out.playNote）。複数行を受ければ重ねて鳴る＝輪唱。
  • 波形表示：従来の FFT スペクトルから、out.mix の時間波形表示に変更（音を視覚化）。
  • 制約：音色合成クラス（AcousticViolinInstrument 等）は変更禁止。今回も無改修。
// violin.pde（受信→発音、抜粋）
if (freq > 0.0 && durMs > 0) {
  out.playNote(0, durMs/1000.0, new AcousticViolinInstrument(freq));
}

9曲データ：カエルの歌

各要素＝8分音符の48スロット・グリッド（休符＝0.0）。チーム正本（Zulip 共有）の楽譜と一致。 四分音符は「音＋休符」、八分音符の連続部はそのまま並べて表現している。

ドレミファミレド (休) / ミファソラソファミ (休)
ド(休) ド(休) ド(休) ド(休)
ドドレレミミファファ / ミレド(休)

コード上は C5=523.2Hz 基準で、配列長 48・ROUND_OFFSET=16（48÷3＝16＝ちょうど1フレーズ）。

10未確定・要チーム調整

① テンポ段階数

実装は3段階（LEVEL:1/2/3）。5段階に戻すならセンサ＋同期側で 1〜5 を扱う改修が必要。

② level→bpm の向き

センサの速度判定が未実装。どの数字が速い/遅いかが決まり次第 bpmTable を合わせる。

③ 輪唱の手動合図

現状は START 後に自動で2声目。任意タイミングで入れるなら新メッセージ（例 ROUND）を親機に追加。

④ サーボの登録

受動リスナーで運用中。正式な子機にするなら親機の namechild／childClients を拡張。

⑤ 音ズレ検証

無線（UDP）の遅延が合奏精度に与える影響を実機で測定し、許容範囲を確認する。
